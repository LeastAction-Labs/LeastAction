<leastAction_requirements>
The payload must be a string and connection which will be a JSON will be passed as parameters.
- least_action_task_object: A dictionary-like object containing task information. Key fields include: laui, last_run_session_id, connection_laui, connection, payload, config, actions, operator_laui, frequency, and logical_date. The object may contain other fields as well.
- all the credentials related values will be present in connection in least_action_task_object.get('connection')
ALL operator code blocks MUST implement exactly 4 methods with the following signatures:
NOTE: All the required parameters will be present in least_action_task_object such as connection, payload, config, laui (task identifier), last_run_session_id etc.
1. initialize(least_action_task_object)
- Purpose: Initialize required clients/connections for the operation
- Parameters:
  - least_action_task_object : least_action_task_object
- Returns: client
- Logging: Use descriptive step names like 'creating_client', 'setting_up_connection'

2. run(least_action_task_object,client)
- Purpose: Execute the main operation/task
- Parameters:
  - least_action_task_object : least_action_task_object
  - client: Client object from initialize() (if applicable)
- Returns: Dictionary containing:
  - Operation-specific results/metadata
  - 'execution_type': 'sync' for synchronous operations (JDBC-like where execution waits for result)
  - 'execution_type': 'async' for asynchronous operations (requires status polling)
  - 'result': result/response/ouput after execting run method.
  - 'status' : 'success/pending/running/error/failed'
- Logging: Log operation start, progress, and completion. log each and every step with data passed if any.

3. check_completion(least_action_task_object,client,run_details)
- Purpose: Check completion status for async operations
- Parameters:
  - least_action_task_object : least_action_task_object
  - client: Client object from initialize() (if applicable)
  - run_details: Results returned from run() method
- Returns: Dictionary with:
  - 'status': One of 'success', 'failed', 'pending'
  - 'message': Descriptive status message
  - 'output': Final results (for successful completions)
- For sync operations: Simply return {'status': 'success', 'message': 'Synchronous operation completed'}
- Logging: Log each status check attempt

4. finish(least_action_task_object,client,completion_details,run_details)
- Purpose: Cleanup resources and close connections
- Parameters:
  - least_action_task_object : least_action_task_object
  - client: Client object to cleanup
  - completion_details: Final completion status
  - run_details: Original run results
- Returns: None
- Logging: Log cleanup steps and any errors
use only standard required parameters for connection and payload unless explicitly specified.
</leastAction_requirements>
