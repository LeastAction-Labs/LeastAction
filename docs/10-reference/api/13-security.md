## 1. Authentication Models

LeastAction uses **dual authentication** to separate user context from system infrastructure validation.

### User Authentication

**Purpose**: Identifies the user making the request and validates their identity.

**Mechanism**: JWT access token passed via:

- Cookie: `frontend_token`
- Header: `Authorization: Bearer <token>`

**Token Contents**:

- User LAUI (`sub`)
- Email
- Account LAUI
- Expiration (24 hours)
- Signature (RS256)

**Generated by**: `POST /api/v1/token` endpoint after successful login or OAuth flow.

**Used for**:

- All user-facing API endpoints
- Audit trails and logging
- Permission checks in Ory Keto

### System Authentication

**Purpose**: Proves that internal API calls originate from trusted system infrastructure (Celery workers, cron schedulers).

**Mechanism**: Long-lived JWT passed via:

- Header: `X-System-Auth-Token: <system_jwt>`

**Token Contents**:

- System user LAUI (`sub`) - matches `system@leastactionlabs.com`
- Long expiration (years)
- Signature (RS256)

**Generated by**: Setup script during initial deployment

**Validated by**: `celery_auth_middleware`

**Used for**:

- Worker-to-backend communication
- Scheduler-to-backend communication
- Internal task state updates
- Connection queue management

### Why Two Tokens?

| Scenario                    | User Token          | System Token                   |
| --------------------------- | ------------------- | ------------------------------ |
| **User creates task**       | ✅ (identifies who) | ❌                             |
| **Worker updates task**     | ✅ (user context)   | ✅ (proves worker is legit)    |
| **Scheduler queries tasks** | ✅ (system user)    | ✅ (proves scheduler is legit) |

This separation ensures:

- User actions are properly attributed
- System operations cannot be triggered by external clients
- Worker requests are cryptographically verified
- Audit logs maintain user context even in async operations

## 2. Authorization Architecture

LeastAction uses **Ory Keto** (a Zanzibar-inspired permission system) for fine-grained access control.

### Permission Model

| Permission         | Description                | Example Use Case                    |
| ------------------ | -------------------------- | ----------------------------------- |
| `own`              | Full control over the item | Delete, modify permissions, edit    |
| `edit`             | Modify item content        | Update task config, change workflow |
| `view`             | Read-only access           | View task details, read payload     |
| `delete`           | Remove item                | Soft-delete item                    |
| `true_parent_edit` | Edit as true parent        | Modify linked child items           |
| `is_true_parent`   | True parent check          | Validate parent-child relationships |

### Access Check Flow

```
Request → Router Dependency → Ory Keto Check → Service Layer
            (Auth middleware)    (Permission)    (Business logic)
```

**Router Dependency**: FastAPI `Depends()` annotations validate permissions before the request reaches the service layer.

**Ory Keto**: External service that stores and evaluates subject-resource-relation tuples.

**Service Layer**: Assumes access has been validated and focuses on business logic.

**Exception**: The `catalog/create` endpoint validates access internally in the service layer to ensure atomic item creation with permission setup.

## 3. System-Only Endpoints

Certain endpoints are **exclusively** accessible by system components (workers, schedulers). External access is blocked via the `celery_auth_middleware`.

### Protected Endpoints

| Endpoint                                                    | Purpose                                   | Caller         |
| ----------------------------------------------------------- | ----------------------------------------- | -------------- |
| `POST /api/v1/task/update/{task_laui}`                      | Update task state during execution        | Celery worker  |
| `POST /api/v1/task/finish/{task_laui}`                      | Dequeue task and free connection capacity | Celery worker  |
| `GET /api/v1/catalog/get/tasks_ready_to_run/{project_laui}` | Query tasks eligible for execution        | Cron scheduler |

### Middleware Validation

The `celery_auth_middleware` intercepts requests to these endpoints and:

1. Checks for the `X-System-Auth-Token` header
2. Validates the JWT signature
3. Verifies the token's `sub` (subject) matches the system user LAUI
4. Rejects requests without a valid system token with `401 Unauthorized`

```python
# Middleware check
if not system_auth_token:
    return 401 Unauthorized - "Missing X-System-Auth-Token"

claims = verify_jwt(system_auth_token)
if claims.sub != system_user.laui:
    return 401 Unauthorized - "Tampered celery auth token"
```

### Why These Endpoints Are Protected

| Endpoint              | Risk if Exposed                                                     |
| --------------------- | ------------------------------------------------------------------- |
| `/task/update`        | External clients could manipulate task state, bypass workflow logic |
| `/task/finish`        | Could prematurely free connection slots, causing race conditions    |
| `/tasks_ready_to_run` | Reveals infrastructure scheduling patterns, creates DoS vector      |

## 4. Item Type Security

### Restricted Item Types

The following item types **cannot** be created via `/api/v1/catalog/create`:

- `task`
- `action`

**Why**: Tasks and actions require specialized validation, permission checks, and lifecycle management.

**How to Create**:

- Tasks: `POST /api/v1/task`
- Actions: `POST /api/v1/action`

**Validation**:

```python
# catalog/create endpoint
if item.item_type in ["task", "action"]:
    raise InvalidArgumentError(
        "Invalid item type passed",
        f"use /api/v1/{item.item_type} api to create {item.item_type}"
    )
```

**Enforcement**: The check happens at the **router level** before reaching the service layer.

### Dedicated Endpoints

#### `/api/v1/task`

**Access Checks**:

- If `item_laui` provided: **edit** on task
- If creating new:
  - **edit** on `parent_laui`
  - **view** on `operator_laui`, `connection_laui`, `payload_laui`
  - **view** on all action LAUIs in `actions` field

**Why**: Tasks can reference multiple resources (operators, connections, payloads, actions). The endpoint validates access to **all** referenced items before creating the task.

#### `/api/v1/action`

**Access Checks**:

- If `item_laui` provided: **own** on action
- If creating new:
  - **edit** on `parent_laui`
  - **view** on `connection_laui` (if provided)

**Why**: Actions execute arbitrary Python code. Requiring **own** permission ensures only the action creator can execute it.

## 5. Router-Level vs Service-Level Checks

### Router-Level Authorization (New Pattern)

**Where**: FastAPI route definitions via `Depends()` annotations

**When**: Before the request reaches the service layer

**Advantages**:

- **Fail fast**: Rejects unauthorized requests immediately
- **Clean separation**: Authorization logic separate from business logic
- **No redundant queries**: Single Ory Keto check at the entry point
- **Consistent**: Same pattern across all endpoints

**Example**:

```python
@task_router.post("")
async def create_run_task(
    request: Annotated[BaseCreateItemRequest, Depends(validate_access_for_create_run)],
    item_orchestrator: ItemOrchestrator = Depends(get_item_orchestrator),
):
    # Access already validated - service layer can focus on business logic
    return await item_orchestrator.create_run_task(request)
```

### Service-Level Authorization (Legacy Pattern)

**Where**: Inside service methods

**When**: After the request reaches the service layer

**Disadvantages**:

- **Mixed concerns**: Business logic and authorization intertwined
- **Redundant checks**: Multiple Keto queries for complex operations
- **Harder to audit**: Access logic scattered across service methods

**Exception**: The `catalog/create` endpoint still validates access in the service layer because:

- Item creation and permission setup must be atomic
- Access checks happen **during** the creation transaction
- Prevents race conditions between creation and permission assignment

**Example**:

```python
# catalog_service.create_item() (internal)
async def create_item(self, request):
    # Access check happens here
    await self.access_reader.check_item_edit_access(parent_laui, user_laui)

    # Create item and set permissions atomically
    async with transaction():
        item = await self.repo.create(request)
        await self.keto_writer.create_access(item_laui, user_laui, "own")
```

### Migration Summary

**Before**: Access checks scattered across service methods

**After**: Access checks centralized at the router level for all endpoints **except** `catalog/create`

**Impact**:

- ✅ Cleaner service layer code
- ✅ Faster rejection of unauthorized requests
- ✅ Easier to audit and test
- ✅ Reduced Ory Keto query load

## 6. Security Best Practices

### For API Consumers

1. **Always include the Bearer token** for user-facing endpoints
2. **Never expose the `X-System-Auth-Token`** - it's for internal use only
3. **Use `/task` and `/action`** - don't attempt to create tasks/actions via `/catalog/create`
4. **Check permissions before operations** - use `/access/get/permission` to validate access

### For Developers

1. **Add access checks at the router level** for new endpoints
2. **Use FastAPI dependencies** - `Depends(validate_access)` pattern
3. **Keep service layer clean** - assume access is validated
4. **Document system-only endpoints** - mark them clearly in API docs
5. **Test permission scenarios** - unauthorized, insufficient permissions, valid access

### For Infrastructure

1. **Protect the system token** - store securely, rotate periodically
2. **Monitor system endpoint access** - alert on unauthorized attempts
3. **Audit token usage** - track which workers use the system token
4. **Separate worker networks** - isolate Celery workers from public internet

## Security Checklist

When adding a new endpoint:

- [ ] Determine if it's user-facing or system-only
- [ ] If system-only, add to `celery_auth_middleware` protected routes
- [ ] Define required permissions (own, edit, view, delete)
- [ ] Implement router-level dependency for access checks
- [ ] Document authentication and authorization requirements
- [ ] Test unauthorized access scenarios
- [ ] Test permission boundary cases (e.g., view vs edit)
- [ ] Verify audit logs capture user context correctly
